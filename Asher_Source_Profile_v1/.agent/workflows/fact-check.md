# /fact-check - 事实核查与知识增密

> **核心原则:** 量大管饱。每一个核查点都是一次知识扩展的机会。

---

## 触发条件

```
/fact-check [文件路径]
```

或在学习模块完成后自动执行。

---

## 执行流程

### Phase 0: 预处理

// turbo
```
1. 读取目标文件
2. 识别文件的:
   - 核心主题 (Theme)
   - 目标受众 (Audience)
   - 预期产出 (Goal)
3. 提取所有可核查的论断 (Claims)
```

**可核查论断识别标准:**
- 事实性陈述 (X 是 Y)
- 因果关系 (因为 X 所以 Y)
- 数据引用 (X% / X 年)
- 定义声明 (X 的定义是 Y)
- 类比/隐喻 (X 像 Y)

---

### Phase 1: 逐点核查 + 内联标注

> [!CAUTION]
> **每个核查点必须同时输出:**
> 1. 核查结论 (✅ 正确 / ⚠️ 部分正确 / ❌ 错误)
> 2. 知识点扩展 (概念定义 + 现实案例)
> 3. 如有错误：指出哪里错了 + 事实是什么

**标注格式:**

```markdown
> 原文论断: "XXXXXX"

**核查结论:** ✅ 验证通过 / ⚠️ 部分正确 / ❌ 需修正

**知识点扩展:**

〔概念名〕**XXXXX**

- **定义:** ...
- **核心机制:** ...
- **学术来源:** ...

〔现实案例〕

**案例 1: [标题]**
- 背景: ...
- 过程: ...
- 结果: ...
- 与本论断的关联: ...

**案例 2: [标题]**
- ...

〔如有错误〕

**❌ 错误指正:**
- **原文说:** ...
- **事实是:** ...
- **差异在于:** ...
- **为什么会错:** ...
- **修正建议:** ...
```

---

### Phase 2: 整体审视 + 洞察生成

> [!IMPORTANT]
> 完成逐点核查后，必须后退一步审视全局。

// turbo
**执行步骤:**

1. **主题一致性检查**
   - 所有论断是否服务于核心主题?
   - 有没有跑题的内容?

2. **逻辑链完整性检查**
   - 论证链条是否存在断裂?
   - 是否有跳跃式推理?

3. **被忽略维度识别**
   
   根据主题和目标，检查是否遗漏了:
   
   | 维度 | 检查问题 |
   |:---|:---|
   | 反面论证 | 是否只说了正面，忽略了反例? |
   | 边界条件 | 在什么情况下这个论断不成立? |
   | 历史脉络 | 这个观点是怎么形成的? 有没有演变? |
   | 竞争观点 | 有没有其他学派/人持不同意见? |
   | 实践维度 | 理论很好，但怎么落地? |
   | 受众盲区 | 读者可能会有哪些误解? |

4. **新洞察生成**

   基于核查过程中发现的知识点，提出:
   - 原文没有说到但值得说的内容
   - 跨领域的类比联想
   - 更深一层的"为什么"
   - 可操作的行动建议

---

### Phase 3: 知识增密

> [!CAUTION]
> **量大管饱原则:** 不是堆砌，而是每个点都要展开到足够深。

**增密标准:**

| 类型 | 最低要求 |
|:---|:---|
| 概念解释 | 定义 + 核心机制 + 为什么重要 |
| 现实案例 | 至少 2 个，覆盖不同场景 |
| 类比/隐喻 | 至少 1 个通俗易懂的比喻 |
| 边界说明 | 什么时候不适用 |
| 行动指引 | 知道这个之后能做什么 |

**增密优先级:**

1. **核心论断** — 必须完整增密
2. **关键转折** — 论证链条的关键节点
3. **反直觉观点** — 需要额外解释才能接受的内容
4. **专业术语** — 任何可能造成理解障碍的词汇

---

### Phase 4: 错误修正专区

> [!WARNING]
> 如果发现错误，必须明确输出以下结构:

```markdown
## ❌ 错误修正记录

### 错误 1: [简述]

**原文表述:**
> "XXXXXX"

**事实真相:**
> "YYYYYY"

**错在哪里:**
- 具体错误点: ...
- 为什么会犯这个错: ...
- 这个错误的影响: ...

**修正方案:**
- 建议改为: "ZZZZZZ"
- 或删除该论断

**补充知识:**
[提供正确的知识点扩展]
```

## 质量标准

| 检查项 | 要求 |
|:---|:---|
| 知识密度 | 每个核查点必须有概念+案例 |
| 案例数量 | 每个核心论断至少 2 个案例 |
| 错误指正 | 不能只说"错了"，必须说清楚事实是什么 |
| 洞察数量 | 至少输出 3 个新洞察 |
| 被忽略维度 | 至少识别 2 个 |
| 输出长度 | 输出应 ≥ 原文长度的 50% |

## 示例: 核查点标注

**原文论断:**
> "高价值资料往往需要前置知识才能理解，而人类在理解之后，会下意识默认'别人也应该懂'。"

**核查结论:** ✅ 验证通过

**知识点扩展:**

〔概念〕**知识的诅咒 (Curse of Knowledge)**

- **定义:** 一种认知偏差，使具有深度专业知识的人难以与知识较少的人有效沟通。
- **核心机制:** 
  - 专家忘记了"不知道"是什么感觉
  - 无意识假设他人拥有相同背景知识
  - 跳过关键基础概念或未提供足够上下文

- **学术来源:** 
  - Camerer, Loewenstein & Weber (1989) "The Curse of Knowledge in Economic Settings"
  - 该术语由经济学家 Robin Hogarth 首创

〔现实案例〕

**案例 1: 软件开发者与用户**
- 背景: 一位资深程序员向产品经理解释技术方案
- 过程: 使用了大量术语如 "API"、"微服务"、"容器化" 而未解释
- 结果: 产品经理无法做出决策，项目延误
- 关联: 展示了专家沟通时的"默认懂"假设

**案例 2: 医生与患者**
- 背景: 医生向患者解释病情
- 过程: 使用医学术语，语速快，没有确认理解
- 结果: 患者回家后不遵医嘱，因为根本没听懂
- 关联: 高知识密度领域普遍存在此问题

〔通俗类比〕
> 就像一个会骑自行车的人试图解释"怎么保持平衡"——他自己早就忘了学骑车时有多难。
